# Contributing to omnistack-agent

Thank you for your interest in contributing! This document covers everything you need to get started.

---

## English

### Prerequisites

- **Node.js ≥ 18** (the build system uses native ESM and `node:test` — no polyfills needed)
- **No npm dependencies to install** — the repo is intentionally zero-dependency. Just clone and go.

### The Golden Rule

> **Never edit files inside `adapters/` by hand.**

The `adapters/` directory is fully generated by the build script. Any manual change will be overwritten the next time `npm run build` runs, and CI will reject PRs where the adapters are out of sync with the source. Always edit `core/` or `knowledge/`, then rebuild.

### Workflow

```
fork → create branch → edit core/ or knowledge/ → npm run build → node --test → open PR
```

Step by step:

1. **Fork** the repository and create a descriptive branch (e.g. `feat/knowledge-graphql`, `fix/core-identity-typo`).
2. **Edit** source files only:
   - `core/` — numbered prompt files that form the agent's reasoning backbone.
   - `knowledge/` — modular Markdown files, one topic per file.
3. **Rebuild** the adapters: `npm run build`
4. **Run tests**: `node --test` — all tests must pass.
5. **Commit** your changes (see commit style below).
6. **Open a Pull Request** against `main`. CI will run `npm run validate` to confirm the adapters match the source exactly.

### Knowledge Module Template

When adding a new knowledge module, use this structure:

```markdown
# <Topic>
> One-line summary + when this matters

## Concepts

## Best Practices

## Patterns & Examples   (real, commented code)

## Common Pitfalls / Anti-patterns

## References            (official docs / standards)

<!-- level: beginner | intermediate | advanced -->
```

Place the file under the appropriate subfolder inside `knowledge/` and register it in `knowledge/_index.md`.

### Commit Message Style

This project uses **Conventional Commits**. Examples from the repo history:

| Scope | Example |
|---|---|
| `docs(knowledge): ...` | Adding or updating a knowledge module |
| `feat(core): ...` | New capability in the core prompt |
| `fix(core): ...` | Correcting existing core content |
| `build(scripts): ...` | Changes to `build.mjs` / `validate.mjs` |
| `ci: ...` | GitHub Actions changes |
| `docs: ...` | README, CONTRIBUTING, CHANGELOG |

Format: `<type>(<optional scope>): <short imperative description>`

### CI

Every pull request runs **GitHub Actions** (`ci.yml`):

1. `node --test` — unit tests must pass.
2. `npm run validate` — confirms all 9 adapters are byte-for-byte in sync with the source.

A red CI check means the PR is not ready to merge.

## Português (PT-BR)

### Pré-requisitos

- **Node.js ≥ 18** — o sistema de build usa ESM nativo e `node:test`; nenhum polyfill é necessário.
- **Sem dependências npm para instalar** — o repositório é intencionalmente zero-dependência. Basta clonar e começar.

### A Regra de Ouro

> **Nunca edite arquivos dentro de `adapters/` manualmente.**

O diretório `adapters/` é totalmente gerado pelo script de build. Qualquer alteração manual será sobrescrita na próxima vez que `npm run build` for executado, e o CI rejeitará PRs em que os adapters estejam fora de sincronia com o código-fonte. Sempre edite `core/` ou `knowledge/`, depois reconstrua.

### Fluxo de Trabalho

```
fork → criar branch → editar core/ ou knowledge/ → npm run build → node --test → abrir PR
```

Passo a passo:

1. **Faça um fork** do repositório e crie uma branch descritiva (ex.: `feat/knowledge-graphql`, `fix/core-identity-typo`).
2. **Edite** apenas os arquivos-fonte:
   - `core/` — arquivos de prompt numerados que formam a espinha dorsal do agente.
   - `knowledge/` — arquivos Markdown modulares, um tópico por arquivo.
3. **Reconstrua** os adapters: `npm run build`
4. **Execute os testes**: `node --test` — todos devem passar.
5. **Faça o commit** das suas alterações (veja o estilo abaixo).
6. **Abra um Pull Request** contra `main`. O CI executará `npm run validate` para confirmar que os adapters correspondem exatamente ao código-fonte.

### Template de Módulo de Conhecimento

Ao adicionar um novo módulo de conhecimento, use esta estrutura:

```markdown
# <Tópico>
> Resumo em uma linha + quando isso importa

## Concepts

## Best Practices

## Patterns & Examples   (código real, comentado)

## Common Pitfalls / Anti-patterns

## References            (docs oficiais / padrões)

<!-- level: beginner | intermediate | advanced -->
```

Coloque o arquivo na subpasta apropriada dentro de `knowledge/` e registre-o em `knowledge/_index.md`.

### Estilo de Mensagem de Commit

Este projeto usa **Conventional Commits**. Exemplos do histórico do repositório:

| Escopo | Exemplo |
|---|---|
| `docs(knowledge): ...` | Adicionando ou atualizando um módulo de conhecimento |
| `feat(core): ...` | Nova capacidade no prompt central |
| `fix(core): ...` | Correção de conteúdo existente no core |
| `build(scripts): ...` | Alterações em `build.mjs` / `validate.mjs` |
| `ci: ...` | Alterações no GitHub Actions |
| `docs: ...` | README, CONTRIBUTING, CHANGELOG |

Formato: `<tipo>(<escopo opcional>): <descrição imperativa curta>`

### CI

Cada pull request executa o **GitHub Actions** (`ci.yml`):

1. `node --test` — os testes unitários devem passar.
2. `npm run validate` — confirma que todos os 9 adapters estão sincronizados byte a byte com o código-fonte.

Uma verificação de CI com falha significa que o PR não está pronto para merge.
